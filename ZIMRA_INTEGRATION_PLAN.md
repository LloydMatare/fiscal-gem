# ZIMRA Fiscal Device Gateway Integration Plan for SaaS

This document outlines the technical architecture and implementation roadmap for building a SaaS application that integrates with the ZIMRA Fiscal Device Gateway (API v7.2).

## 1. System Architecture & Multi-Tenancy

The core challenge for a SaaS fiscalization app is managing **many fiscal devices**, each with its own unique identity and security credentials.

### 1.1. Certificate Management (CRITICAL)
Your current implementation in `lib/fdms/client.ts` uses `process.env.FDMS_CLIENT_CERT_PEM`. This works for **one** device. For a SaaS, you must manage a unique certificate **per tenant/device**.

*   **Database Schema**: You need a `devices` table to store:
    *   `id`: Internal UUID.
    *   `serialNumber`: Manufacturer serial number (e.g., "SN001").
    *   `deviceId`: ZIMRA issued ID (e.g., 1234567890).
    *   `activationKey`: One-time key from ZIMRA portal.
    *   `certificate`: The X.509 certificate (PEM) returned by `registerDevice`.
    *   `privateKey`: The private key generated by your app (encrypted!).
    *   `mode`: 'ONLINE' | 'OFFLINE'.
    *   `status`: 'REGISTERED' | 'ACTIVE' | 'EXPIRED'.

*   **Security**:
    *   **NEVER** store the `privateKey` in plain text. Use a symmetric encryption key (e.g., AES-256) stored in environment variables (or a KMS like AWS KMS/Google Secret Manager) to encrypt the private key before saving to the DB.
    *   Decrypt the key only in memory when performing a transaction.

### 1.2. Dynamic Client Instantiation
You must refactor `lib/fdms/client.ts`. Instead of a singleton `fdmsClient` initialized with env vars, export a factory function:

```typescript
// Proposed signature
export function createFdmsClient(deviceCert: string, deviceKey: string) {
  return {
    submitReceipt: async (data: ReceiptData) => { /* ... uses specific cert/key ... */ },
    // ... other methods
  };
}
```

## 2. Integration Roadmap

### Phase 1: Onboarding & Registration (The "Handshake")
Before a user can sell, they must register their device.

1.  **Generate Keys & CSR**:
    *   Use `node-forge` or `crypto` to generate an RSA-2048 or ECC key pair.
    *   Create a CSR (Certificate Signing Request).
    *   **Format**: CN must be `ZIMRA-<Serial>-<DeviceID>`.
    *   **Validation**: This runs *on your server*.

2.  **Register Device**:
    *   Endpoint: `POST /api/v7.2/registerDevice` (No mTLS required for this call).
    *   Input: `deviceID`, `activationKey`, `certificateRequest` (the CSR).
    *   Output: `certificate` (save this!).

### Phase 2: Configuration & Daily Workflow
Once registered, the device enters the daily cycle.

1.  **Get Config**:
    *   Call `getConfig` using the new certificate (mTLS).
    *   Store `taxRates` and `taxPayerDetails` in your DB to ensure your UI calculates taxes exactly as ZIMRA expects.

2.  **Open Fiscal Day**:
    *   **Requirement**: Must be done once every 24 hours (max).
    *   Endpoint: `openDay`.
    *   Action: Create a record in your `fiscal_days` table.

3.  **Issue Receipts**:
    *   **Signature Generation**: You must sign specific fields of the receipt.
    *   **QR Code**: Generate a URL based on the response.
    *   **Chaining**: Each receipt must include the hash of the *previous* receipt. You need to store the `lastReceiptHash` in your `devices` table.

4.  **Close Fiscal Day**:
    *   **Aggregation**: Sum up all sales/taxes for the day.
    *   Endpoint: `closeDay`.
    *   Signature: Requires signing the daily totals.

### Phase 3: Offline Handling (Queueing)
If your SaaS supports offline mode (or if ZIMRA is down):

1.  **Queue**: Store signed receipts in a `pending_receipts` table.
2.  **Batch**: A background job (e.g., BullMQ, Inngest, or Cron) checks for connectivity.
3.  **Submit**:
    *   If Online: Retry `submitReceipt`.
    *   If Offline Mode: Generate a JSON file, base64 encode it, and use `submitFile`.

## 3. Technical Specifics

### 3.1. Data Models (TypeScript/Zod)
You need robust types for the complex `submitReceipt` payload.

```typescript
interface ReceiptItem {
  description: string;
  quantity: number;
  price: number; // in cents
  taxCode: string; // "A", "B", etc.
  total: number; // in cents
}

// ... expanded types needed for strict compliance
```

### 3.2. Signature Generation Logic
This is the most complex part. The signature is a `Base64(Sign(Hash(Normalize(Data))))`.

*   **Normalize**: Concatenate fields strictly (no separators).
    *   Receipt: `deviceID + type + currency + globalNo + date + total + taxes + prevHash`
    *   **Critical**: Decimal formatting rules (e.g., `15.00` for tax percent).
*   **Hash**: SHA-256.
*   **Sign**: RSA-SHA256 or ECDSA-SHA256 using the device's private key.

## 4. Immediate Next Steps

I recommend we implement the following in order:

1.  **Refactor Client**: Update `lib/fdms/client.ts` to support dynamic certificates.
2.  **Crypto Utils**: Create `lib/fdms/crypto.ts` to handle:
    *   Key pair generation.
    *   CSR creation (using `node-forge` or similar).
    *   Device Signature generation (normalization + signing).
3.  **Database**: Update your schema (Drizzle?) to store device credentials securel.

## 5. Security Checklist
- [ ] **Private Keys**: Encrypted at rest.
- [ ] **Environment**: `ENCRYPTION_KEY` set in `.env`.
- [ ] **Logs**: Ensure `privateKey` or raw `receiptSignature` inputs are never logged.
- [ ] **Backups**: Database backups must include the encrypted keys (otherwise devices are orphaned).
